# Jira Test Data Benchmark

Generate synthetic Jira data and benchmark Fivetran's Jira connector for BQ-DTS.

## Problem

We need to evaluate Fivetran's Jira connector — throughput, object coverage, data integrity — before building our own DTS connector. Real customer Jira data can't be used for benchmarks: it's sensitive, small, and not reproducible. We need a controlled, synthetic dataset that covers all 20 Fivetran-synced Jira object types at scale (100K+ issues), with known distributions and edge cases baked in.

## How It Works

The suite generates three datasets inside a real Jira Cloud instance, then validates and benchmarks them:

```
config.py                  Central config: projects, archetypes, volumes, auth
    │
    ├── generate_contract.py   Coverage dataset (5K issues, all 20 objects)
    ├── generate_csv.py        Scale dataset CSVs → Jira bulk import (100K issues)
    ├── augment.py             Enrich via API: comments, worklogs, transitions, links, sprints
    ├── edge_cases.py          Stress dataset (12 categories, ~1,400 adversarial issues)
    │
    ├── verify.py              Quick smoke test: are all 20 objects populated?
    └── eval_suite.py          Full validation (B1-B6, Q1-Q9) + benchmark manifest
```

**Shared utilities** (`utils/`):
- `rate_limiter.py` — Thread-safe Jira API client with 429 retry, backoff, and connection timeouts
- `text_generator.py` — Realistic ticket titles, descriptions, comments, and edge case text
- `distributions.py` — 5 project archetypes with field-level distributions

### Three Datasets

| Dataset | Script | Volume | Purpose |
|---------|--------|--------|---------|
| **Coverage** | `generate_contract.py` | 5K issues | Systematically populate all 20 Fivetran objects |
| **Scale** | `generate_csv.py` + `augment.py` | 100K issues | Measure throughput and sync performance |
| **Stress** | `edge_cases.py` | ~1,400 issues | Break things: 32K descriptions, 100+ comments, CJK/emoji, deep changelogs |

### Five Project Archetypes

| Key | Archetype | Issues (100K) | Board Type |
|-----|-----------|---------------|------------|
| BENCH-S1 | SCRUM-1 | 30K | Scrum |
| BENCH-S2 | SCRUM-2 | 20K | Scrum |
| BENCH-K1 | KANBAN-1 | 20K | Kanban |
| BENCH-K2 | KANBAN-2 | 15K | Kanban |
| BENCH-CL | CLASSIC | 15K | Classic |

Each archetype has distinct distributions for issue types, priorities, statuses, story points, due dates, comments, and worklogs. Defined in `utils/distributions.py`.

### The 20 Fivetran-Synced Objects

Defined in `config.FIVETRAN_OBJECTS` (single source of truth). Mapped to Jira REST API endpoints:

| # | Object | Jira API | Generated By |
|---|--------|----------|-------------|
| 1 | Issues | `/rest/api/3/issue` | `generate_csv.py`, `generate_contract.py` |
| 2 | Issue Field History | changelog (via transitions) | `augment.py` (transitions pass) |
| 3 | Issue Multiselect History | changelog (multi-select fields) | `generate_contract.py` |
| 4 | Comments | `/rest/api/3/issue/{key}/comment` | `augment.py` (comments pass) |
| 5 | Worklogs | `/rest/api/3/issue/{key}/worklog` | `augment.py` (worklogs pass) |
| 6 | Issue Links | `/rest/api/3/issueLink` | `augment.py` (links pass) |
| 7 | Watchers / Votes | `/rest/api/3/issue/{key}/watchers` | `generate_contract.py` |
| 8 | Issue Properties | `/rest/api/3/issue/{key}/properties/{key}` | `generate_contract.py` |
| 9 | Remote Links | `/rest/api/3/issue/{key}/remotelink` | `generate_contract.py` |
| 10 | Projects | `/rest/api/3/project` | Pre-configured in Jira |
| 11 | Components / Versions | `/rest/api/3/component`, `/rest/api/3/version` | `generate_contract.py` |
| 12 | Boards | `/rest/agile/1.0/board` | Pre-configured in Jira |
| 13 | Sprints | `/rest/agile/1.0/sprint` | `generate_contract.py`, `augment.py` |
| 14 | Users | `/rest/api/3/users/search` | Pre-configured in Jira |
| 15 | Groups | `/rest/api/3/group/bulk` | Pre-configured in Jira |
| 16 | Atlassian Teams | Atlassian Admin (not REST) | Pre-configured |
| 17 | Fields & Field Options | `/rest/api/3/field` | Pre-configured + custom fields |
| 18 | Issue Types / Statuses / Priorities | `/rest/api/3/issuetype`, etc. | Pre-configured |
| 19 | Project Roles & Permission Schemes | `/rest/api/3/project/{key}/role` | Pre-configured |
| 20 | Security Schemes & Levels | `/rest/api/3/issuesecurityschemes` | Pre-configured (optional) |

## Priority Framework

| Priority | What | Pass Criteria |
|----------|------|---------------|
| **Must-pass (B checks)** | All 20 objects populated, references valid, timestamps ordered, no dead links | Binary pass/fail. Benchmark results cannot be published if any B check fails. |
| **Quality (Q checks)** | Distribution accuracy, comment/worklog counts, edge case coverage, tail numbers | Proximity to target. Advisory — results can be published with deviations noted. |
| **Calibration (Q9)** | Synthetic vs real Jira comparison | Informational only. Measures the gap between declared priors and production data. |

## Setup

### Prerequisites

- Python 3.10+
- A Jira Cloud instance with 5 projects pre-configured (BENCH-S1 through BENCH-CL)
- Jira API token with admin access
- Scrum and Kanban boards configured for the relevant projects

### Install

```bash
git clone https://github.com/jack-c379/jira-test-data-benchmark.git
cd jira-test-data-benchmark
pip install -r requirements.txt
cp .env.example .env
# Fill in JIRA_URL, JIRA_EMAIL, JIRA_API_TOKEN
```

### Quick Validation (Tiny Profile)

To quickly verify config and connectivity without generating 100K issues:

```bash
python generate_csv.py --scale-factor 0.01    # 1K issues (CSVs only, no API calls)
python generate_contract.py --dry-run          # Preview coverage plan
python verify.py                               # Check what's already in Jira
```

## Usage

### Phase 0: Generate Data

```bash
# Step 1: Coverage dataset (5K issues via API — all 20 objects)
python generate_contract.py

# Step 2: Scale dataset (100K issues via CSV → import via Jira UI)
python generate_csv.py
# Then: Jira > System > Import > CSV — import each output/scale_*.csv

# Step 3: Enrich scale data with related objects
python augment.py                    # All 5 passes (comments, worklogs, transitions, links, sprints)
python augment.py --pass comments    # Or run passes individually

# Step 4: Stress dataset (1,400 edge case issues)
python edge_cases.py
```

### Phase 1: Validate

```bash
python verify.py              # Quick smoke test (all 20 objects populated?)
python eval_suite.py          # Full eval (B1-B6, Q1-Q9) + manifest generation
python eval_suite.py --checks B1,Q1,Q2  # Specific checks only
```

### Scaling

All scripts respect `SCALE_FACTOR` in `config.py`:

| Factor | Total Issues | Use Case |
|--------|-------------|----------|
| 0.01 | 1K | Config validation, quick smoke test |
| 0.1 | 10K | Development iteration |
| 1.0 | 100K | Standard benchmark |
| 5.0 | 500K | Scale stress test |
| 10.0 | 1M | Full throughput benchmark |

Override per-run: `python generate_csv.py --scale-factor 0.1`

Note: At very small scales (factor << 0.1), all counts are clamped to minimum 1. Relative distributions may be distorted.

## Eval Checks

### B1-B6: Basic (Must Pass)

| Check | What | Pass Criteria |
|-------|------|---------------|
| B1 | All 20 objects populated | Every object >= 1 row |
| B2 | References valid | 0/1000 broken references |
| B3 | Timestamps ordered | updated >= created, resolutiondate >= created |
| B4 | Issue links valid | Zero dead link targets |
| B5 | Sprint assignments valid | Zero orphan sprint references |
| B6 | Resolved = terminal status | Done/Closed/Resolved only |

### Q1-Q9: Quality (Advisory)

| Check | What | Target |
|-------|------|--------|
| Q1 | Total issue count | 100K +/- 5% |
| Q2 | Issue type distribution | Within 10% per project |
| Q3 | Priority distribution | Within 10% per project |
| Q4 | Field coherence | No impossible field combinations |
| Q5 | Comment count | Avg 3/issue +/- 20% |
| Q6 | Worklog count | Avg 0.8/issue +/- 20% |
| Q7 | Edge cases present | All 12 categories found |
| Q8 | Tail numbers | Max comments >= 100, changelog >= 20 |
| Q9 | Synthetic vs real | Informational (provide --real-sync-file) |

## Rollout

### Local Development

1. Clone repo, install deps, fill `.env`
2. Run with `--scale-factor 0.01` or `--dry-run` to verify
3. Generate at full scale, validate with `eval_suite.py`

### CI Integration

1. Add `.env` vars as CI secrets
2. Run `generate_csv.py --scale-factor 0.01` + `verify.py` as a pre-merge check
3. Full benchmark runs on a schedule (weekly) against a dedicated Jira instance

### Multi-Instance

The suite is parameterized via `.env`. To benchmark against different Jira instances, use separate `.env` files:

```bash
cp .env .env.staging
# Edit .env.staging with staging Jira credentials
DOTENV_PATH=.env.staging python generate_csv.py
```

### Updating the Object List

If Fivetran adds or removes synced objects, update `config.FIVETRAN_OBJECTS` — this is the single source of truth. Then update `generate_contract.py` to ensure the new objects are populated, and `verify.py`/`eval_suite.py` to check them.
